CPU Utilization

It is crucial to pay attention to CPU resource allocation on the cluster. When you adjust the application's CPU resources, consider the following strategies for effective management:

  1. Aim to set the requested cpu (request.cpu) at around 80% of what your application uses, plus an additional 20%. For example, if your application uses 50m from Monday to Friday, 9:00 a.m. to 5:00 p.m, set the (request.cpu) value to 60m
  2. Avoid setting the requested cpu below 10m or 20m, as there is limited value in going too low, and it may have some drawbacks
  3. Be mindful of multithreaded applications or multi-process containers. Threads and processes can consume quota more quickly, resulting in application throttling
  4. Remember that the application’s performance is impacted by the CPU limit (limit.cpu) and not by the requested cpu (request.cpu)
 

Disk Storage

With OpenShift 4.x, you can expand Persistent Volume Claims (PVC). You start with limited disk storage but it is possible to increase it as needed.

To monitor PVCs effectively, use SysDig to keep track of capacity and alert your team when it reaches at around 80%. You can then adjust the size to accommodate future demands. If you need help setting up SysDig alerts for your PVCs, please reach out to Billy Li for assistance.

 

Memory

Applications usually require a specific amount of memory and it's often challenging to reduce this requirement. Here are essential strategies to manage memory resources effectively:

  1. Once your application has been running for a while and you understand its memory usage pattern, adjust the "request.memory" value to be approximately 20% more than what your application needs.
  2. Use the “limit.memory” setting as a tool for detecting memory leaks.
  3. Avoid running your application consistently above the "request.memory" value as it could lead to the application being moved between nodes when they adjust their workloads.
